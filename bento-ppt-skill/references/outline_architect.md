# 大纲架构参考

这份参考来自你现有的“PPT 结构架构师”思路，但收敛成 skill 内可复用的版本。

目标只有一个：

把主题、上下文和约束，变成一个逻辑严密、层次清晰、可继续下游编排的 `PPT_OUTLINE` JSON。

## 什么时候读这个文件

- 用户只有主题，没有完整大纲。
- 用户给了一堆材料，但没有明确章节结构。
- 用户想先确认 PPT 逻辑，再决定视觉和版式。

## 大纲方法

默认使用金字塔原理：

1. 结论先行
2. 上位概括下位
3. 同层内容要同类归组
4. 按明确逻辑展开：时间、空间、因果、对比、流程、问题到解法

如果用户给了背景调研、搜索结果或最新事实，必须纳入大纲判断，避免“主题是新的，内容却还是旧模板”。

## 写大纲前的最小输入

最好能拿到这些信息：

- `topic`
- `audience`
- `scenario`
- `duration_or_page_range`
- `context`
- `must_include`
- `must_avoid`
- `language`

缺失时可合理假设，但要在结果后注明假设。

## 页数建议

### 5-7 页

适合 5-10 分钟快讲。

建议结构：

1. 封面
2. 问题或背景
3. 核心观点一
4. 核心观点二
5. 证据或案例
6. 行动建议
7. 结尾

### 8-12 页

适合常规业务汇报、产品介绍、提案陈述。

建议结构：

1. 封面
2. 目录
3. 背景 / 关键判断
4-5. 第一部分
6-7. 第二部分
8-9. 证据 / 数据 / 案例
10-11. 方案 / 建议
12. 结尾

### 12-18 页

适合策略汇报、研究综述、路演。

建议结构：

1. 封面
2. 目录
3. 总论点
4-6. 第一部分
7-9. 第二部分
10-12. 第三部分
13-15. 数据 / 案例 / 实证
16-17. 建议 / 下一步
18. 结尾

### 18-25 页

适合深度分析、培训、完整方案讲解。

控制原则：

- 每一部分 3-5 页即可
- 每 3-4 页必须有节奏变化
- 不要把“目录、过渡、总结”都删掉，否则观众会失去定位感

## 标准输出格式

结果必须用 `[PPT_OUTLINE]` 和 `[/PPT_OUTLINE]` 包裹，并严格输出 JSON。

```json
{
  "ppt_outline": {
    "cover": {
      "title": "主标题",
      "sub_title": "副标题",
      "content": []
    },
    "table_of_contents": {
      "title": "目录",
      "content": ["第一部分", "第二部分"]
    },
    "parts": [
      {
        "part_title": "第一部分：章节标题",
        "pages": [
          {
            "title": "页面标题",
            "content": ["要点一", "要点二"]
          }
        ]
      }
    ],
    "end_page": {
      "title": "总结与展望",
      "content": ["结论一", "结论二"]
    }
  }
}
```

## 内容写法约束

- 每页只保留一个主命题。
- 页标题要能单独成立，不能只是“背景介绍”“方案分析”这种空词。
- `content` 里的条目应该是页面要点，不是整段演讲稿。
- 如果上下文包含关键数据、时间点、政策名、公司名，优先落进具体页面，而不是只出现在总标题里。

## 建议输出策略

如果用户要先确认方向，先交付两部分：

1. 一段 3-6 句的“整体叙事说明”
2. `PPT_OUTLINE` JSON

如果用户直接要求继续做页级编排，再进入 `page_system.md`。

## 常见错误

- 用“行业现状、解决方案、未来展望”机械套壳，不结合主题。
- 大纲每页都像一级标题，没有真实论点。
- 把研究摘要原文复制成 `content`。
- 不区分“章节页”和“内容页”，导致后面难以编排。
